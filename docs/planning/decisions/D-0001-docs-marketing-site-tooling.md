# Documentation & marketing site tooling

## Summary

- Ship the docs + marketing site as **one Astro site with Starlight**, not two separate sites — Astro is the only Node-toolchain option that treats marketing and docs as first-class in one project. See [[#Decision]].
- Keep docs content behind a clean `src/content/docs` seam so a future extraction into a standalone docs site stays cheap. See [[#Consequences]].
- Generate **API reference from TSDoc via TypeDoc**, mounted into the Starlight sidebar (`starlight-typedoc`) — one site, autogenerated from code. See [[#Decision]].
- API pages render **through Starlight's theme**, so they inherit the site's styling for free; polish is Starlight CSS, not a separate TypeDoc theme. See [[#Why]].
- Defer a separate marketing property or standalone `/api` deploy until concrete pressure appears. See [[#Out of scope]].

^summary

## Context

This project is a Node/TypeScript template. Projects generated from it need a
public website that does two jobs: explain the project (marketing/landing) and
document it (guides plus API reference). The author has been using Astro
Starlight. The open questions were: (1) one combined site or two separate sites
— weighing a pure API/code-reading docs tool against a broader marketing site;
(2) which framework(s) to standardize on as the template default; and (3) how to
produce — and style — API reference generated from the code.

## Decision

The template ships a **single Astro site**. Starlight provides the documentation
surface; ordinary Astro pages provide the marketing/landing surface in the same
project. API reference is generated from TSDoc comments by **TypeDoc** and
mounted into the Starlight sidebar via **`starlight-typedoc`** (Markdown output,
not TypeDoc's standalone HTML). Docs content lives behind a clean
`src/content/docs` boundary so the docs surface can later be extracted into its
own site without a rewrite. No separate marketing site or standalone API deploy
ships today.

## Why

- **Astro is the only Node/TS-toolchain option that treats marketing and docs as first-class in one framework.** Every pure-docs tool (VitePress, Docusaurus, Material for MkDocs, mdBook) makes marketing pages an afterthought. One framework → one build, deploy, domain, and design system → the lowest-maintenance default for a template.
- **"Pure API/code reading" is a pipeline concern, not a reason to split sites.** TypeDoc emits Markdown that mounts into the same site, so autogenerated API reference needs no second property.
- **Styling comes for free from the Markdown pipeline.** Because `starlight-typedoc` produces Markdown that Starlight renders, the API pages inherit the site theme — dark mode, sidebar, search, typography — instead of TypeDoc's dated standalone HTML. Further polish is ordinary Starlight theming (custom CSS / props / Tailwind), applied once across docs and API alike. TypeDoc's own HTML theme would have needed a separate theme (e.g. Material/DMT) or bespoke CSS to look comparable.
- **A marketing/docs split only pays off under pressure** — non-engineer marketing ownership, a release cadence divorced from the code, or an API reference large enough to warrant its own versioning/deploy. None is present in a template default, so paying the two-codebase, two-design-system cost now is premature. The clean content seam keeps the split cheap if it later becomes warranted.
- Staying on Astro also matches the author's existing experience; the decision is the strong technical fit for the combined goal, not just incumbency.

## Options considered

### Site architecture

- **One unified site, split-ready (chosen)** — marketing + docs in one Astro project, docs behind a clean seam. Lowest maintenance; a split is deferred but cheap.
- One unified site, permanent — same, but with no designed seam. Rejected: closes the split door for little gain.
- Two separate sites from day one — cleaner audience separation and independent deploys, but two codebases and two design systems to keep consistent. Rejected as premature for a template default.

### Docs framework

- **Astro + Starlight (chosen)** — does both jobs; MDX, Pagefind search, i18n, autogenerated sidebar, strong Lighthouse scores.
- Docusaurus — most mature, versioned docs, built-in blog; heavier, React-only, marketing secondary.
- VitePress — fastest and most minimal; weak marketing story (hero layout only).
- Fumadocs — modern DX, strong API-reference support; newer/less battle-tested and ties to Next.js.
- Material for MkDocs / mdBook — excellent docs but off-toolchain (Python / Rust) for a Node template.

### API reference

- **TypeDoc → Starlight sidebar (chosen)** — autogenerated from TSDoc, one site, inherits the site theme.
- TypeDoc standalone `/api` HTML — separate deploy needing its own theme/CSS; unnecessary unless the API is the primary product.
- No autogen for now — defer; rejected because an API reference is a core reason to document a library/template.

## Consequences

- New template dependencies: `astro`, `@astrojs/starlight`, `starlight-typedoc`, `typedoc`.
- Docs authors write MDX under `src/content/docs`; marketing authors edit Astro pages under `src/pages`; both share one theme.
- API-page styling is governed by Starlight theming (custom CSS / component overrides / Tailwind), not a TypeDoc theme — one styling surface for the whole site.
- Keeping the `src/content/docs` boundary clean becomes a maintenance rule; a future split depends on it.
- One static-output deploy pipeline covers both surfaces.
- Delivered by [[C-0001-docs-marketing-site]] (status `open/planned` until built).

## Out of scope

- A separate marketing property (CMS-driven, conversion-optimized) — revisit if marketing ownership or cadence diverges from the code.
- A standalone versioned API site — revisit if the API surface warrants independent versioning/deploy.
- Hosting/deploy target (Cloudflare Pages, Netlify, GitHub Pages) — a later, separable decision.

## Open questions

- Does the marketing landing live at the site root with docs at `/docs`, or do docs live at the root? (affects the seam and the split path)
- Deploy target and CI wiring for the static build.
- How much custom Starlight styling to invest up front vs. shipping the stock theme and refining later.
- Whether to adopt Starlight's versioned-docs approach now or defer until the API stabilizes.
